# Football Decision Engine

本项目的长期目标不是做一个简单的“体彩分析网站”，也不是停留在“让 GPT 写一篇赛前分析”的层面。

我们真正要搭建的是一个 **足球比赛决策引擎（Football Decision Engine）**。

前端页面只是展示层。真正有长期价值的是底层引擎：它能持续接入比赛数据、沉淀历史样本、提取可复用特征、记录模型判断、自动回填赛果，并通过复盘不断修正模型。

当这个引擎成熟后，它不只服务体彩赛事分析，也可以扩展到比赛研究、媒体内容生产、数据 API、模型服务，甚至开放给其他产品调用。

## 项目定位

本项目面向足球比赛的赛前决策、赛后验证和长期模型迭代。

它的核心不是“预测一场比赛”，而是建立一套可以长期复盘、长期积累、长期升级的判断系统。

当前世界杯和体彩赛事页面，是这个决策引擎的第一阶段应用场景。

## 长期价值

项目价值来自以下资产的持续叠加：

- 数据资产：赛程、盘口、赔率、比分、赛果、锁版记录和复盘结果。
- 特征工程：赛事类型、联赛风格、球队状态、盘口结构、赔率变化、赛程动机、进球区间、让球风险。
- 赔率演化：记录开盘、临场、SP 漂移、压热压冷和跨市场冲突。
- 历史样本：沉淀相似比赛、相似盘口、相似模型版本和相似错误类型。
- AI 解释：把模型判断过程结构化，而不是只生成一段文字分析。
- 锁版验证：赛前判断必须保存，赛后不能倒推修改。
- 持续复盘：按联赛、赛事类型、模型版本、盘口类型和错误原因持续优化。

这些内容叠加起来，才是平台真正的护城河。

## 核心闭环

项目必须围绕下面这条闭环建设：

```text
赛事接入
  -> 数据清洗
  -> 特征提取
  -> 模型推演
  -> 人工锁版
  -> 赛果自动回填
  -> Case Base 样本生成
  -> 相似案例调用
  -> 复盘诊断
  -> 模型升级
```

每一次比赛结束，都应该让样本库更有价值，而不是只更新一个比分。

## 统一模型底座

所有联赛和杯赛模型都必须以世界杯 V4 推演链为基础，不能另起一套只看赔率的简化模型。

版本号按赛事独立维护：世界杯当前为 V4；新联赛从本联赛 V1 开始，但 V1 也必须复用世界杯 V4 的核心推演顺序：

```text
内部实力 / 概率底盘
  -> 赛事规则与动机
  -> 近期状态
  -> 风格与战术对位
  -> 机构线与体彩盘口偏差
  -> 赔率动态防守层
  -> 常规比赛脚本
  -> 半场 / 60分钟触发脚本
  -> 决策冲突闸门
  -> 比分与总进球校验
  -> 让球独立闸门
  -> 失败方式识别
  -> 价值过滤
  -> 人工确认后锁版
```

联赛模型补齐阶段只能写 `PRE_LOCK`。只有完成球队状态、风格对位、赛事规则、相似案例、半全场分支、比分/总进球、让球闸门、失败方式和价值过滤后，才允许人工确认升级为 `FINAL_LOCK`。

2026-07-04 复盘后，联赛 V1 额外执行硬门槛：

- **球队状态门槛**：`teamState` 必须写出双方排名/近3-5场/主客场/伤停轮换/进攻路径/防守风险。只有“结合联赛节奏、主客场和盘口热度”的模板文本不能进入 `FINAL_LOCK`。
- **赔率动态门槛**：`lineMovement` 必须比较开盘、最新、临场或 SP 历史中至少两个状态；没有赔率动态数据时只能降级，不允许升到 `B-` 以上。
- **冲突闸门**：单独一个 `1-1` 最低比分不能推翻胜平负低位方向。只有球队状态、联赛画像、赔率动态、让球映射、相似样本中至少两层共同支持，才允许改选平局或冷门。
- **让球映射门槛**：两个候选比分必须逐个换算让胜/让平/让负，不能把“受让保护”直接等同于 `让胜`。
- **总进球门槛**：低比分模板不能自动导出 `1/2球` 或 `2/3球`；必须结合联赛画像和双方近期 BTTS/大球形态。

## 外部历史样本

新联赛或样本不足的联赛，优先通过免费数据源或 API 拉取历史赛果，统一写入 `web/data/externalHistoricalSamples.js`。

外部历史只作为 `external-history / EXTERNAL_HISTORY` 参考样本进入相似案例引擎，用来校验同赛事的胜平负分布、平局率、进球区间和常见比分。它不能直接生成锁版，也不能直接进入 D1 Case Base。

Case Base 只接收我们自己的 `FINAL_LOCK -> 赛果回填 -> 复盘诊断` 之后生成的样本。这样可以避免把外部赛果误当成模型命中记录。

通用 API 导入入口：

```bash
npm run history:import-api -- --provider football-data --competition-code CODE --season 2026 --league 瑞超 --api-key "$FOOTBALL_DATA_API_KEY"
npm run history:import-api -- --provider thesportsdb --league-id ID --season 2026 --league 瑞超
npm run history:import-api-football -- --leagues 瑞超,韩职,韩职2 --seasons 2026,2025 --limit 100 --api-key "$APISPORTS_API_KEY"
```

赔率和盘口样本入口：

```bash
npm run history:import-football-data -- --leagues 英超,西甲,德甲,意甲,法甲,荷甲,葡超,瑞超,挪超,芬超,丹超,日职,中超,美职
npm run history:import-football-data -- --input-dir ./tmp/football-data-csv --leagues 瑞超,挪超,芬超,丹超,日职,中超,美职
```

`history:import-football-data` 使用 Football-Data.co.uk CSV，写入 `external-history` 赔率/盘口样本。它会提取 1X2、大小球 2.5、亚洲盘口让球线和主客水位，用于补强 `盘口画像`，但仍然不生成锁版、不进入 D1 Case Base。

## 样本资产目标

下一阶段的重点，是确认每一场赛果是否能自动保存为可复用样本。

样本不是只保存“主队 2-1 客队”这种结果，而是要尽量保存下一场模型可以调用的上下文。

每条样本至少应包含：

- 基础信息：赛事、联赛、轮次、主队、客队、开赛时间、体彩期号。
- 赛前盘口：胜平负、让球、总进球、比分赔率、开盘和临场变化。
- 赛前模型：模型版本、单选方向、让球方向、总进球、比分预测、置信等级、建议动作。
- 推演依据：赛事权重、球队状态、风格对位、盘口偏差、比赛脚本、半场触发、风险排除。
- 实际赛果：全场比分、半场比分、胜平负、让球结果、总进球、比分是否覆盖。
- 复盘结果：命中项、未命中项、失败原因、是否降级、是否进入相似案例库。
- 样本标签：联赛类型、比赛类型、盘口类型、进球区间、强弱关系、冷门/平局/打穿风险。

只有保存这些字段，样本达到 1000 场、10000 场时，才真正能反哺模型，而不是变成一堆不可用的比分记录。

## 存储与扩展要求

项目的数据层必须按长期积累设计。

需要重点验证：

- Cloudflare D1 当前容量是否足够支撑 1000 场、10000 场样本。
- 每场样本的结构化字段和 JSON payload 大小是否可控。
- 是否需要把大文本推演、赔率历史、逐分钟事件拆成独立表。
- 是否需要区分热数据和冷数据：近期比赛用于页面快速读取，历史样本用于相似案例和模型训练。
- 是否需要定期导出到 CSV / JSON / 对象存储，作为二级备份和离线研究数据。

原则：短期用最简单可运行的 D1 表结构，长期预留迁移到更大样本库的路径。

## 下一阶段优先级

1. 确认赛果自动回填是否稳定。
2. 确认每场已锁版比赛是否会自动生成 Case Base 样本。
3. 检查样本字段是否足够支持下一场相似案例调用。
4. 检查 D1 存储容量、表结构和索引是否能支撑长期积累。
5. 把复盘页从“结果展示”升级为“模型诊断中心”。
6. 按联赛建立独立模型版本：世界杯 V4，新增联赛从各自 V1 开始，但全部复用世界杯 V4 推演链。
7. 将前端展示降级为结果层，把数据、模型、样本、复盘作为项目核心。

## 关键原则

- 赛前锁版才算数。
- 赛后只复盘，不倒推修改旧判断。
- 样本必须结构化，不能只保存文字。
- 模型必须可解释，但解释不能替代数据。
- 前端页面服务于引擎，不反过来限制引擎。
- 长期目标是提高判断质量，不是追求每场都有结论。
